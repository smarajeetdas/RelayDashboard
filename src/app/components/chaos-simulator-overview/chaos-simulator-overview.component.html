Overview

A resilient system detects failure and routes around it. Non-resilient systems collapse when faced with a malfunction. In contrast, resilient systems are responsive to failure and take a proactive stance to put in place mechanisms, that expect and survive failure. Failures can happen due to network issues, load & capacity mismatch or configuration errors. We need to decouple such component failures from outages that are visible to users.

Chaos Engineering allows for better, more advanced resilience testing and enhanced customer experience. Developers can experiment in cloud-native distributed systems, wherein the experiments involve testing, both the physical infrastructure and the cloud ecosystem.

Chaos Testing Lifecycle

01

Architecture Assessment

  • Objective of chaos testing
  • Identify environment & tools to perform chaos
  • Evaluate System Architecture and identify target systems
02

Build Hypothesis

  • Perform load test strategy to evaluate the steady state metrics
  • Identify failure to inject, and hypothesize what might go wrong
  • Create chaos test cases & define what to test next
03

Run Experiment

  • Ensure proper alerting is in place for critical metrics
  • Notify teams on chaos testing schedule and impact
  • Start with minimized blast radius
04

Validate Hypothesis

  • Analyze the results to conclude if the hypothesis is proved
  • Identify if system is resilient to the real-world events
  • Document findings and observations
05

Results & Proposition

  • Create tickets on resiliency improvement items
  • Publish the gamedays results
  • Provide recommendations on improvement

Why is it Important?

{{ i }}

Get visibility into weak components of the system

Chaos Engineering allows you to identify hidden weaknesses before they cause system-wide failures by systematically testing your infrastructure's ability to withstand unexpected conditions.

Improve MTTD / MTTR

Reduce Mean Time To Detect (MTTD) and Mean Time To Repair (MTTR) by running controlled experiments that help teams practice response procedures and improve their ability to quickly recover from failures.

Increase Resilience of the system

Build more robust and fault-tolerant systems by continuously testing against various failure scenarios, ensuring your architecture can adapt to unexpected conditions.

Reduce system and application downtime

Prevent costly service disruptions by proactively discovering and addressing vulnerabilities in your system architecture before they cause production incidents.

Minimize revenue loss due to downtime

Protect your business from financial impact by reducing service outages and ensuring critical systems remain available even under stress conditions.

Improve User Experience

Enhance customer satisfaction by ensuring systems degrade gracefully during partial failures and maintain core functionality even when individual components fail.

Prepare the system against unexpected failures

Build confidence by systematically introducing controlled failures and validating that your system can handle real-world scenarios such as service outages, network problems, and resource constraints.

Improve confidence in system

Give teams greater assurance in the reliability of their systems by demonstrating they can withstand turbulent conditions, allowing for more aggressive innovation and faster release cycles.

Chaos Engineering Capabilities

Chaos Engineering
Infrastructure

CPU

Generates high load for one or more CPU cores.

Memory

Allocates a specific amount of RAM.

I/O

Puts read/write pressure on I/O devices such as hard disks.

Disk

Writes files to disk to fill it to a specific percentage.

Shutdown

Performs a graceful shutdown (and an optional reboot) on the host operating system.

AI

GPU

Test that your application behaves as expected even when GPU capacity is limited or exhausted.

Network

Latency

Injects latency into all matching egress network traffic.

Black hole

Drops all matching network traffic.

Packet Loss

Induces packet loss into all matching egress network traffic.

DNS

Blocks access to DNS servers.

ALFI
(Application-Level Fault Injection)
